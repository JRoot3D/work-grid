# Decision Log

- **Single phase, not two.** The team lead suggested a possible two-phase split (file/symbol rename → importer updates). Rejected because every viable split leaves a transient broken-build state: if phase 1 renames the file, all 5 importers fail to resolve `'../state/boardSlice'` until phase 2 lands. That violates the planner-rule "each phase is independently revertable and leaves the codebase shippable" (`.claude/rules/teams-flow/planner.md`). Total expected delta is ~9 line edits + 1 file rename — comfortably under the ~150-line phase guideline, so a single phase is the correct shape.

- **Tool choice for symbol renames.** Per `_shared.md` should-fix rule, Serena's `rename_symbol` is preferred for whole-symbol renames. It atomically renamed `boardReducer` → `workspaceReducer` and (within the same file) `boardSlice` → `workspaceSlice`. **Caveat:** `rename_symbol` does NOT rename files or update import-path strings — the file rename and the 5 path-string updates were separate mechanical edits. Order followed: (1) rename file with `git mv` to preserve history, (2) `rename_symbol` for the two consts inside the renamed file, (3) update the 5 import-path strings in importers.

- **Directory rename: Option A (leave as-is) over Option B (move to `src/features/workspace/`).** Locked at brief time. Rationale: the user-visible feature is "the board"; workspace is an implementation detail of that feature's state. Option B would create churn in `git history` for tidiness on paper without user-visible benefit, and would require a thin re-export shim. Option A keeps the diff minimal.

- **Doc sweep deferred.** Stale `boardSlice.ts` references in `docs/`, `.claude/rules/teams-flow/`, and `.flow-spec/project.md` were left untouched. Locked at brief time. Rationale: the "smallest possible change" senior-baseline rule wins; doc references rot but are non-blocking. Reviewer/checker explicitly do not flag them. A separate one-line docs sweep can ship later as its own brief.
